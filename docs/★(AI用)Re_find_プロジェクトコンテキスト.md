# Re:find — プロジェクトコンテキスト

> **このファイルの使い方**
> チームメンバーが生成AI（ChatGPT、Claude、Copilotなど）を使うとき、会話の最初にこのファイルを読み込ませてください。
> AIがプロジェクトの目的・方針・技術構成を理解した上で回答するようになり、チーム全体でAI出力の質を揃えることができます。

---

## 1. Re:find とは何か

Re:find は「情報再発見」サービスです。

人はネットで気になる情報を見つけると、スクショやブックマークで保存します。
しかし保存した情報は他の情報に埋もれ、忘れられ、結局行動に移せないまま終わります。

Re:find は、この **「保存して終わり」を「保存して、再び出会える」に変える** サービスです。

---

## 2. Re:find が解決する課題

### 根幹にある問題

人がその人にとって価値あることに時間を使えていない。

隙間時間はある。でも必要な情報が出てこないから動けない。
情報の **散逸**・**再検索**・**先延ばし** —— これらが原因で、保存した情報が行動につながらない。

### 具体的な困りごと（ユーザーインタビューに基づく）

- スクショが大量の写真に埋もれて見つからない
- ブックマークが溜まるだけで読まない
- LINEの会話に流されて情報が見つからない
- 同じ情報を何度も検索し直す（時間のムダ）
- 夫婦で何度も議題に上がるのに、毎回進まない
- 「あとでやろう」が「やらないまま」になり、後悔する

### ターゲットユーザー

「後でやろう」と思って情報を保存したまま、埋もれて忘れ、行動に移せずにいる人。
保存した情報を活かしきれず、隙間時間も有意義に使えていないと感じている人。

### 対象となる情報の性質（アイゼンハワーマトリクス）

Re:find が扱うのは **「重要だが緊急ではない」情報** です。

- 子どもの習い事の候補（調べたけど比較・申し込みまでいかない）
- あとで読みたい記事（保存したけど読まない）
- 買い物の候補（見つけたけど購入判断を先延ばし）
- 返信すべきメッセージ（急ぎではないが大事）

緊急なものはその場で対処する。重要でないものは忘れても困らない。
問題は **重要なのに緊急じゃないから、ずっと後回しにされ続ける** こと。

## 3. Re:find の守備範囲（何をする / 何をしない）

### ✅ Re:find がやること

| やること | 説明 |
|---------|------|
| 情報の一箇所集約 | 種類を問わず（スクショ、URL、テキスト）LINEに送るだけで保存 |
| 自動整理 | AIが内容を理解し、タイトルとカテゴリを自動でつける |
| 再発見の機会提供 | 忘れた頃にリマインド通知を送り、情報を再び目の前に届ける |
| 見返しやすい整理 | Webアプリでカテゴリ別にタイル表示。すぐ見つかる状態を維持 |

### ❌ Re:find がやらないこと

| やらないこと | 理由 |
|-------------|------|
| 行動を強制・保証すること | リマインドしても行動するかは本人次第。「行動させます」とは約束しない |
| タスク管理 | 既知のタスクを管理・実行支援するのは既存アプリ（Todoist等）の領域 |
| モチベーション管理 | ゲーミフィケーションや習慣化の仕組みは作らない |
| 情報の要約・分析 | AIは分類に使うが、内容の深い分析や意思決定の支援はしない |

### Re:find の立ち位置

```
情報を見つける → 保存する → 埋もれる → 再発見する → やることを決める → 行動する
                              ↑                        ↑
                          ここが問題             ここから先は本人が決める

                ├── Re:find の守備範囲 ──┤
                保存 → 埋もれさせない → 再発見

                                              ├── 既存タスク管理アプリ ──┤
                                              やることを管理 → 実行を支援
```

**Re:find は「選択肢をテーブルに戻す」サービス。行動するかどうかはその人が決める。**

## 4. エレベーターピッチ

- ［急ぎではないけれど大事な情報を「後でやろう」と保存したまま、埋もれて忘れ、行動に移せずにいる状況を変え］たい
- ［保存した情報を活かしきれず、隙間時間も有意義に使えていないと感じている人］向けの、
- ［Re:find］というプロダクトは、
- ［埋もれた情報をちょうどいいタイミングで思い出させてくれる「情報再発見」サービス］です。
- これは［保存した情報を自動で整理し、その人が動けるタイミングで届けることで、大事な情報を埋もれさせず、行動に移すきっかけを提供すること］ができ、
- ［情報の種類ごとに別々のアプリを使い分けたり、自分でタスクを言語化して登録する従来のやり方］とは違って、
- ［情報の種類を問わず一箇所に集約でき、整理もリマインドも自動で行うことで、保存した情報を埋もれさせない仕組み］が備わっている。

## 5. 主な機能（Phase 1 スコープ）

### やること

1. **LINE Botでの情報受け取り** — スクショ・URL・テキストをLINEに送るだけで保存
2. **AIによる自動分類・タイトル生成** — 送られた情報の内容をAIが理解し、カテゴリとタイトルを自動でつける
3. **Webアプリでの一覧表示** — カード型タイル表示 + カテゴリフィルタのタブ切り替え
4. **LINEリマインド通知** — 忘れた頃に「この情報、まだ見てないよ」と通知（間隔：1日→3日→7日→14日→30日→60日）。エビングハウスの忘却曲線とSuperMemoの最適間隔を参考に、通知疲れを考慮して設計。最終通知（6回目）では経過日数と通知回数を明示し、「これが最後のリマインドです」と伝える。ユーザーはWebアプリから「未対応」に戻して通知サイクルを再開可能
5. **多人数対応** — LINEログインで自分のデータだけ見える仕組み
6. **アイテムのステータス管理** — 未対応 / 対応済み / アーカイブ
7. **カテゴリ管理** — AI自動生成 + 手動修正 + 名前変更・削除
8. **共有リンクの発行** — アイテムごとにURLを発行、ログイン不要で閲覧可能

### やらないこと（Phase 1）

- ネイティブアプリ化（iOS / Android）
- ユーザー間の共有管理機能（共有はリンク発行のみ）
- カテゴリの高度な管理（並び替え・統合・アイコン設定）
- スマート通知（SNS利用タイミング検知）
- 複雑な検索・並び替え機能
- LINE Bot上での編集操作
- 動画・音声・PDFへの対応（PDFは進捗次第で検討）

## 6. 技術スタック

### 一覧

| 役割 | 技術 | 費用 |
|------|------|------|
| バックエンド | Flask（Python） | ¥0（Render） |
| フロントエンド | HTML + Bootstrap 5 + JavaScript | ¥0 |
| データベース | Supabase（PostgreSQL） | ¥0（無料枠） |
| 画像ストレージ | Supabase Storage | ¥0（無料枠） |
| ユーザー認証 | Supabase Auth（LINEログイン） | ¥0 |
| AI分類 + 画像解析 | OpenRouter API（無料モデル・マルチモーダルLLM） | ¥0 |
| 入力・通知 | LINE Messaging API | ¥0（月200通まで） |
| ホスティング | Render | ¥0（無料枠） |

**運用コスト合計：¥0**

### 各技術の役割

**Flask** — バックエンド。LINE Webhook受信、AI分類リクエスト、OGP取得、Supabaseへのデータ保存、HTML配信、通知Cron Job。

**HTML + Bootstrap 5 + JS** — フロントエンド。FlaskがHTMLを配信し、JSでSupabaseからデータを動的に取得・表示。Bootstrap 5はCDN1行で導入。

**Supabase** — DB（PostgreSQL）+ 認証（Auth）+ 画像保存（Storage）のセット。RLSで「自分のデータだけ見える」をDB側で強制。

**OpenRouter API** — マルチモーダルLLMでスクショを直接理解。OCR不要。1回のAPI呼び出しでタイトル生成 + カテゴリ分類が完了。

**LINE Messaging API** — 入口と通知口。受信は無制限、プッシュ通知は月200通（1人1日1通にまとめて節約）。

**Render** — Flaskホスティング。GitHub連携で自動デプロイ。Cron Job無料枠あり。15分アクセスなしでスリープする制約あり。

### 共有リンクのRLS対応

**方法A（Flask側で処理）を採用。** 共有リンクへのアクセスはFlaskがservice_role_keyでRLSバイパスする。RLSルールは「自分のデータだけ」のままシンプルに保てる。

## 7. データベース設計

### テーブル構成（7テーブル）

```
users               ← LINE ID ↔ Supabase Auth ID の紐づけ
categories          ← ユーザーごとのカテゴリ（UNIQUE: line_user_id + name）
items               ← 保存情報の本体（ソフトデリート方式）
shared_links        ← 共有リンクのトークン管理
user_activity_logs  ← 行動ログ（Phase 1は記録のみ、分析は将来）
user_settings       ← ユーザー設定（Phase 1はデフォルト固定）
notification_rules  ← 通知ルール（Phase 1は間隔固定：1,3,7,14,30,60日）
```

Storage: `screenshots/{line_user_id}/{item_id}.png`

### 主要テーブルのカラム

**items**（メインテーブル）
- `id` UUID（PK）、`line_user_id` TEXT、`type` TEXT（image/url/text）
- `original_url` TEXT、`image_url` TEXT、`title` TEXT、`description` TEXT
- `category_id` UUID（FK → categories, ON DELETE SET NULL）
- `memo` TEXT、`status` TEXT（pending/done/archived）、`ogp_image` TEXT
- `next_notify_at` TIMESTAMPTZ、`notify_count` INTEGER
- `created_at`、`updated_at`（トリガー自動更新）、`deleted_at`（ソフトデリート）

**categories**
- `id` UUID（PK）、`line_user_id` TEXT、`name` TEXT、`created_at`
- `UNIQUE (line_user_id, name)` — 同名カテゴリ重複防止

### 重要な設計ルール

**⚠️ ソフトデリート：** itemsにアクセスする**すべての箇所**で `deleted_at IS NULL` を条件に含めること。

```python
# ✅ 正しい
.eq("line_user_id", uid).is_("deleted_at", "null")

# ❌ 間違い — 削除済みも取得される
.eq("line_user_id", uid)
```

**カテゴリ削除時：** ON DELETE SET NULLで category_id がNULLになった後、Flask側で「未分類」カテゴリに付け替える。

**service_role_key：** RLSを無視できる管理者キー。**絶対にフロントエンド（JS）に書かない。** Flaskサーバー側のみで使う。

## 9. データの流れ（代表例）

### スクショを送ったとき

```
① LINE Bot → Flask（Webhook受信）
② LINE APIから画像をダウンロード
③ Supabase Storageに画像アップロード → image_url取得
④ OpenRouter APIに画像送信 → AIがタイトル＋カテゴリ名を返す
⑤ categoriesテーブルで既存カテゴリ確認 → あれば使う / なければ新規作成
⑥ itemsテーブルにINSERT（next_notify_at = 翌日21時）
⑦ LINEに返信「🎹 子どもの習い事 に保存しました！」
```

### リマインド通知（毎日21時）

```
Cron Job → notify.py
① next_notify_at ≦ 現在 AND status='pending' AND deleted_at IS NULL を取得
② ユーザーごとに1通にまとめてLINEプッシュ送信
③ notify_count +1、次の通知日時を計算（1日→3日→7日→14日→30日→60日→archived）
④ 最終通知（6通目）は経過日数・通知回数を明示し「最後の通知です」と伝える
```
